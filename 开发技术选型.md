## 2. 各层技术选型详解

### 2.1 前台 UI 层

| 技术 | 选型 | 理由 |
|------|------|------|
| **框架** | Next.js 14+ (App Router) | SSR/SSG 支持、SEO 友好、React 生态成熟；Agent 搜索结果页需要 SSR 提升首屏速度 |
| **语言** | TypeScript | 类型安全、前后端共享类型定义、.amd Schema 可直接映射为 TS 接口 |
| **样式** | Tailwind CSS | 原子化 CSS、极简风格开发效率高、与 Google 风格 UI 契合 |
| **状态管理** | Zustand | 轻量、无样板代码、适合搜索状态（query/filters/results）管理 |
| **搜索交互** | Monaco Editor (轻量集成) | 为高级用户/开发者提供 .amd 语法高亮的在线编辑器 |
| **可视化** | D3.js / React Flow | 展示快照 DAG 分支树、Agent 协作拓扑图 |
| **API 通信** | tRPC 或 REST + SWR | 类型安全的 API 调用，SWR 实现搜索结果缓存与增量更新 |

### 2.2 API Gateway / 后端服务层

| 技术 | 选型 | 理由 |
|------|------|------|
| **主服务框架** | Go (Gin / Fiber) | 高并发、低延迟；与 Eino 框架同语言，Harness 层集成零摩擦 |
| **辅助服务** | Python (FastAPI) | AI/ML 生态丰富，向量 Embedding 生成、CRA 审查脚本等用 Python 更高效 |
| **API 协议** | gRPC + REST (双协议) | gRPC 用于内部服务间高吞吐通信，REST 用于外部 Agent SDK 接入 |
| **认证** | 基于公钥签名的 DID 认证 | Agent ID 即公钥哈希，请求签名验证身份，无需中心化 OAuth |
| **限流与熔断** | Redis + 令牌桶算法 | 防止恶意 Agent 滥用搜索 API，与 Token 经济体联动 |

### 2.3 搜索引擎核心层

| 技术 | 选型 | 理由 |
|------|------|------|
| **向量数据库** | Milvus (主选) / Qdrant (备选) | Milvus: 开源、支持分布式部署、十亿级向量检索、混合查询（向量+标量过滤）；Qdrant: Rust 编写，单机性能极强，适合 MVP |
| **Embedding 模型** | BGE-M3 (多语言) / text-embedding-3-large | BGE-M3: 开源、支持中英文、可本地部署保数据主权；text-embedding-3-large: 精度更高，Cloud_Allowed 场景使用 |
| **全文检索引擎** | Elasticsearch / Meilisearch | ES: 成熟稳定，支持复杂标量过滤与聚合；Meilisearch: 轻量、Rust 编写，适合 MVP 快速启动 |
| **排序算法** | 自研 AgentRank | 基于以下加权因子的综合排序：|

**AgentRank 排序因子:**

```
Score = w1 × SemanticSimilarity
      + w2 × StructuralCompleteness   (步骤拆解细致度)
      + w3 × AbstractionQuality        (I/O Schema 定义清晰度)
      + w4 × ReuseCount                (被复用/克隆次数)
      + w5 × SuccessRate               (历史成功率)
      - w6 × TokenCost                 (Token 消耗惩罚)
      + w7 × CertifiedBoost            (官方认证快照提权)
```

**冷启动策略:**

系统上线初期（索引量 < 100,000），动态调整权重：
- `w2` (StructuralCompleteness) 提升 2.5x：优先推荐步骤拆解完整的快照
- `w1` (SemanticSimilarity) 中 Domain_Tags 匹配权重提升 3x
- `w4` (ReuseCount) 权重降低至 0.3x：避免新发布链路因无复用记录而沉底
- 引入 `CertifiedBoost`：官方预置的标准 SOP 库（如 WMS 同步、TMS 运费比价）获得初始高分
- 随索引量增长，权重自动回归标准配比

### 2.4 .amd 解析器与 Blueprint Parser

| 技术 | 选型 | 理由 |
|------|------|------|
| **YAML 解析** | Go: `gopkg.in/yaml.v3` / Python: `PyYAML` | 解析 Frontmatter 元数据 |
| **Markdown 解析** | Go: `goldmark` / Python: `markdown-it-py` | 解析 Markdown Body 的 CoT 链路结构 |
| **Schema 验证** | JSON Schema (Draft 2020-12) | 校验 .amd 文件的 Input/Output Contract 格式合法性 |
| **Blueprint Parser** | LLM-based (Gemini 2.5 Pro / GPT-4o) | 将 .amd Markdown Body 转化为 Eino StateGraph JSON 配置 |
| **编译缓存** | PostgreSQL JSONB + Redis | 首次翻译后固化 StateGraph JSON，后续直接命中缓存，避免重复 LLM 调用 |
| **Fallback 解析器** | 自研规则引擎 (Go) | LLM 服务不可用时，回退到基于正则+模板的轻量解析器 |

**Blueprint Parser 编译缓存架构:**

```
[.amd 文件入库]
       │
       ▼
[检查缓存: Task_CID 是否已编译?]
       │
       ├─► 命中 → 直接返回 StateGraph JSON (0 Token, <5ms)
       │
       └─► 未命中 → 调用 LLM 翻译
                        │
                        ▼
               [JSON Schema 校验]
                        │
                        ├─► 合法 → 写入 PostgreSQL JSONB + Redis 缓存
                        │
                        └─► 不合法 → 重试 (最多 2 次) 或标记为解析失败
```

**关键设计点:**
- Task_CID 作为缓存 Key，内容哈希天然保证缓存一致性
- 新分支创建时，仅对差异部分触发增量重编译
- 缓存容量预估：100 万快照 × 平均 5KB StateGraph = ~5GB，完全可内存化

### 2.5 存储与网络层

| 技术 | 选型 | 理由 |
|------|------|------|
| **分布式存储** | IPFS (Kubo 实现) | 内容寻址（CID）天然契合 .amd 的 Task_CID 设计；去中心化、抗审查 |
| **P2P 网络** | libp2p | IPFS 底层网络库，支持 DHT (Kademlia)、NAT 穿透、多协议传输 |
| **关系型数据库** | PostgreSQL 16 | 存储 Agent 注册信息、分支关系图、支付流水、CRA 审查记录 |
| **缓存层** | Redis 7 (Cluster) | 热点快照缓存、搜索 Session 状态、分布式锁 |
| **对象存储** | MinIO | 存储 .amd 原始文件、Embedding 向量备份、沙盒测试产物 |
| **消息队列** | NATS / Kafka | NATS: 轻量、Go 原生、适合 Agent 间实时消息；Kafka: 高吞吐日志流，适合索引增量更新 |

### 2.6 Harness 执行引擎

| 技术 | 选型 | 理由 |
|------|------|------|
| **图编排框架** | Eino (Go) | 原生 StateGraph、Interrupt/Checkpoint、StatePreHandler/StatePostHandler，与 Drew 架构高度契合 |
| **备选框架** | LangGraph (Python) | LangChain 生态，Python 开发者友好，支持类似的图状态机能力 |
| **沙盒运行时** | WASM (Wasmtime) / Docker | WASM: 毫秒级启动、强隔离、适合轻量 API 验证；Docker: 完整隔离，适合复杂环境 |
| **契约测试** | 自研 Contract Validator | 基于 .amd 的 Input/Output Contract，自动生成测试用例并校验 Schema 对齐 |
| **适配器生成** | LLM → 纯代码 (Python/JS) | 翻译 Agent 一次性生成映射脚本，注入 StatePostHandler，零运行时 Token 消耗 |

### 2.7 CRA (Code Review Agent) 网关

| 技术 | 选型 | 理由 |
|------|------|------|
| **安全扫描** | Semgrep + 自研 Prompt Injection Detector | Semgrep: 静态代码分析、自定义规则；自研检测器专门针对 LLM 生成的代码进行投毒检测 |
| **模糊测试** | Atheris (Python) / go-fuzz | 自动生成边界测试用例，验证补丁分支健壮性 |
| **沙盒执行** | gVisor / Firecracker MicroVM | 比 Docker 更强的安全隔离，防止恶意代码逃逸 |
| **工作流引擎** | Temporal | 编排 CRA 三阶段审查流水线，支持长时运行、重试、状态回溯 |

### 2.8 支付结算层（双通道架构）

| 技术 | 选型 | 理由 |
|------|------|------|
| **Web3 支付网络** | Lightning Network (BTC) / Solana | Lightning: 微支付成熟、毫秒结算；Solana: 高 TPS、低 Gas、适合高频分账 |
| **智能合约** | Solidity (EVM) / Rust (Solana) | 锁定押金、自动分账、成功证明验证 |
| **状态通道** | Raiden Network / 自研 | 高频 Agent 间微支付，避免每笔交易上链 |
| **钱包 SDK** | ethers.js / web3.py | Agent 集成钱包能力，哈希域名即收款地址 |
| **企业计费网关** | Stripe + 自研计费引擎 | API 预付费/月结模式，标准发票生成，合规财务走账 |
| **积分模拟系统** | 自研 (PostgreSQL + Redis) | MVP 阶段统一积分体系，Phase 3 可无缝切换至 Web3 或法币通道 |

## 3. MVP 阶段技术栈精简方案

为快速验证核心闭环，MVP 阶段采用精简技术栈：

```
┌─────────────────────────────────────────────────┐
│              MVP 精简技术栈                       │
├─────────────────────────────────────────────────┤
│                                                 │
│  前台: Next.js + TypeScript + Tailwind CSS      │
│    │                                            │
│  API: Go (Gin) REST API                         │
│    │                                            │
│  搜索: Qdrant (单机) + BGE-M3 Embedding         │
│    │                                            │
│  存储: PostgreSQL + 本地文件系统                  │
│    │                                            │
│  解析: Go YAML + goldmark (.amd Parser)          │
│    │                                            │
│  执行: Eino StateGraph (基础版)                   │
│                                                 │
└─────────────────────────────────────────────────┘
```

**MVP 暂不实现:**
- IPFS/libp2p 分布式网络（用中心化存储替代）
- Lightning/Solana 支付（用积分系统模拟）
- gVisor 沙盒（用 Docker 替代）
- Temporal 工作流（用简单状态机替代）

## 5. 技术风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|--------|
| 向量数据库十亿级性能瓶颈 | 搜索延迟升高 | Milvus 分片 + 分区索引；热点缓存；分层检索（先标量过滤后向量搜索）|
| LLM Blueprint Parser 解析不稳定 | StateGraph 生成错误 | 强制 JSON Schema 输出校验；Fallback 为规则解析器；编译缓存避免重复调用 |
| Blueprint Parser 高并发下成本爆炸 | Token 费用失控 | 编译缓存层确保每个 Task_CID 仅翻译一次；缓存命中率目标 >95% |
| IPFS 网络延迟与可用性 | 快照获取慢 | 本地缓存 + CDN 层；关键快照预热 |
| 恶意 Agent 投毒 | 索引污染 | CRA 三阶段审查；社区举报机制；签名验证 |
| 微支付 Gas 费 > 支付金额 | 经济模型崩溃 | 状态通道批量结算；链下计算 + 链上验证；保留企业法币通道备选 |
| AgentRank 冷启动新快照沉底 | 优质内容无法被发现 | 动态权重调整；官方认证快照提权；StructuralCompleteness 权重前置 |
| Eino 框架成熟度 | 功能缺失 | 保持 LangGraph 备选路径；核心抽象层解耦 |
