| 层级 | 技术 | 状态 | 说明 |
|------|------|------|------|

- **Go ↔ Python 接口未打通**：matcher.py 目前是本地关键词搜索，未调用 Go 的 `/api/v1/search`
- **DAG 执行引擎未验证**：executor.py 需要接入真实工具（Shopify CLI、Vercel API 等）
- **QA 验收自动化不足**：qa.py 只有框架，缺少具体检查实现
- **前端无动态数据**：index.html 是静态 mock，未对接后端 API
- **数学模型未参数化**：expectation_model.py 公式已写，但缺少可交互的模拟界面
- **快照 Schema 双轨制**：`.amd`（Go 层）和 `.yaml`（Python 层）尚未统一映射

## 三、Phase 1 详细任务清单（当前优先）

### 3.1 任务 1：Go ↔ Python 接口打通（优先级：P0）

**目标**：让 Python 层的 `matcher.py` 调用 Go 后端的 `/api/v1/search`，而非本地关键词匹配。

**实现步骤**：
1. 在 `src/core/matcher.py` 中新增 `GoSearchClient` 类
2. 配置环境变量 `DREW_GO_API_URL=http://localhost:8080`
3. 将用户查询文本通过 HTTP POST 发送到 `/api/v1/search`
4. 解析 Go 返回的 JSON，提取快照列表 + 相似度分数
5. 在 Python 层叠加商业过滤（price、budget、tags）
6. 回写 `matcher.py`，删除本地 `keywords` 硬编码逻辑

**关键代码模式**：
```python
# src/core/matcher.py
class GoSearchClient:
    def __init__(self, base_url: str = os.getenv("DREW_GO_API_URL", "http://localhost:8080")):
        self.base_url = base_url

    def search(self, query: str, filters: dict = None) -> list[SnapshotMatch]:
        resp = requests.post(f"{self.base_url}/api/v1/search", json={
            "query": query,
            "filters": filters or {},
            "top_k": 10
        })
        return [SnapshotMatch(**item) for item in resp.json()["results"]]
```

**验收**：`python -m src.cli.drew search "跨境电商独立站"` 返回的结果来自 Qdrant 向量搜索，而非本地字符串匹配。

### 3.3 任务 3：DAG 执行引擎端到端 Demo（优先级：P0）

**目标**：`executor.py` 能完整执行一条快照的所有步骤，区分自动节点和人工节点。

**实现步骤**：
1. 定义 `Step` 状态机：`pending` → `running` → `auto_complete` / `waiting_human` → `complete` / `failed`
2. 自动节点：调用对应工具（先 mock，如 `print("[AUTO] 执行 Shopify 初始化...")`）
3. 人工节点：暂停执行，输出提示信息，等待用户输入（CLI 交互）
4. 实现执行日志记录（JSON Lines 格式，便于追溯）
5. 支持变量替换（`{{platform}}` → `Shopify`）
6. 支持步骤间数据传递（前一步输出作为后一步输入）

**CLI 交互示例**：

## 六、关键技术决策记录

### 决策 1：.amd vs .yaml 双轨制如何处理？

**结论**：保持 `.amd` 作为 Go 层的存储格式（已有 1000+ 测试数据），Python 层新增 `.yaml` 作为商业层输入格式。通过 `snapshot.py` 中的 `convert_yaml_to_amd()` 和 `convert_amd_to_yaml()` 双向转换。

**理由**：不破坏现有 Go 解析器，同时让工匠/企业更易读写 YAML。

### 决策 2：Go 后端 vs Python 商业层的职责边界

| 职责 | Go 层 | Python 层 |
|------|-------|-----------|
| 向量搜索 | ✅ 主责 | 调用 Go API |
| 标量存储 | ✅ 主责 | 读取 Go API |
| 快照 CRUD | ✅ 主责 | 代理或补充 |
| DAG 执行 | ❌ | ✅ 主责 |
| QA 验收 | ❌ | ✅ 主责 |
| 数学模型 | ❌ | ✅ 主责 |
| 前端 API | 提供基础 | 提供商业聚合 |

### 决策 3：前端技术栈

**结论**：Phase 1-2 保持纯 HTML/CSS/JS（无框架），Phase 3 视情况迁移到 Vue/React。

**理由**：快速验证交互逻辑，避免框架依赖拖累迭代速度。

## 七、风险点与应对

| 风险 | 影响 | 概率 | 应对 |
|------|------|------|------|
| Go ↔ Python 接口延迟高 | 搜索响应慢 | 中 | 加本地缓存（Redis），或合并为单体服务 |
| 外部 API（Shopify/Vercel）不稳定 | DAG 执行失败 | 高 | 实现重试机制 + 降级策略（mock 模式） |
| 快照 Schema 频繁变更 | 数据不兼容 | 中 | 版本号管理 + 迁移脚本 |
| 向量维度不匹配 | 搜索失效 | 低 | 统一 1024 维，嵌入服务版本锁定 |
| 数学模型参数失真 | 商业决策错误 | 中 | 先用 1 个月真实数据校准，再推广 |
